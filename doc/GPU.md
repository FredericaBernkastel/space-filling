Porting all ADF computation to GPU is possible but is a redesign, not an optimization — and the part that resists (structural tree mutation) is not where the parallel win lives. Porting the read-only ascent phase of find_max_parallel is very feasible, is the natural first target, and current data structures are already unusually GPU-shaped. However, the batch size is capped by an algorithmic limit, not a compute limit, so "much larger batch" only pays off with a small change to how batches are validated.

The one structural obstacle: primitives are closures

Everything in a bucket is an Arc<dyn Fn(P2<F>) -> F>. A GPU cannot call host closures, so any port needs the primitives reified as data: a tagged shape descriptor (Circle{c,r}, NGon{n,...}, affine wrapper, boolean node…) interpreted by the kernel, plus parameters in a flat buffer. For the built-in shapes.rs set that's mechanical — they're all closed-form. For arbitrary user primitives (the Mandelbrot DE) you lose full generality, but the crate already contains its own escape hatch: example 07's baked SDF — rasterize a mask, exact EDT, bilinear sampling, certified L = 1+√2. On GPU that's literally a texture fetch. So the primitive story becomes: analytic shape IR for built-ins, baked distance textures for everything exotic. That covers the library's whole demonstrated surface without asking users to write WGSL.

What's already GPU-shaped

The flat arena is nearly a GPU data structure as-is: Vec<Node> with 64-byte, power-of-two-stride nodes and index-based child links (Option<NonZeroU32>) uploads directly as a storage buffer with zero pointer fixup. Buckets become (offset, count) into a flat primitive-index array. A point query — descend root→leaf, min-reduce over ≤β+1 primitives — is short, has low divergence, and is exactly the inner loop both the ascent and the BnB share. The design decisions you made for lock-free CPU parallelism (single allocation, index links, read-only decide pass + sequential commit) are the same ones a GPU port wants.

find_max_parallel on GPU: yes, with two caveats

One thread (or subgroup lane) per seed running the LineSearch::optimize loop: ~20–60 iterations of 3 field evaluations each, data-dependent accept/reject, early exit. Warp divergence is real (iteration counts differ, tree-descent paths differ) so expect maybe 30–60% SIMD efficiency — irrelevant next to the throughput jump from batch 32-on-4-cores to batch 10⁴–10⁶ on a GPU. The ascents are read-only against one snapshot, so there is no synchronization hazard at all; upload traffic is small (a 1M-circle tree is ~10 MiB of nodes and append-mostly, so per-batch deltas are cheap).

Caveat 1 — precision. Consumer GPUs do f64 at 1/32–1/64 rate. In f32, Δ = 1e-9 is meaningless (f32 has ~1e-7 relative epsilon), and the crate's own goal of sub-2⁻¹⁶ precision dies. The right pattern is hybrid: coarse f32 ascent on GPU to h < ~1e-5, then re-refine only the survivors (a few dozen per batch) on CPU in f64 against the true closures. That keeps the exactness story intact — the GPU is just a very good seed-improver, and final commits are still validated by the sound CPU path.

Caveat 2 — the batch-size ceiling is algorithmic. Dedup keeps a maximum only if its free ball is disjoint from every kept one, all measured against a single snapshot. The number of survivors per batch is therefore bounded by the number of distinct pockets the current field has — early in a fill that's small, so a 100k-seed batch mostly rediscovers the same few maxima and throws the rest away. Late in a fill (millions of tiny pockets) huge batches survive fine. So the batch size should scale with fill state — leaf count is a ready proxy. Additionally, if you want to pipeline (ascend batch k+1 on the GPU while the CPU commits batch k), staleness is almost free to fix: before committing a maximum found against an old snapshot, re-evaluate g(m) ≥ d once against the live tree — a single query — and drop it if a newer shape invaded its ball. That one check makes overlapped GPU/CPU execution sound.

Full ADF on GPU: feasible, poor return

insert_where's read-only decide pass (per-leaf BnB, sdf_geq_everywhere, per-child prune) could run as a work-queue kernel — the BnB stack is bounded by prune_subdiv and fits in registers/local memory, and arena growth maps onto atomic bump allocation, which current arena layout practically begs for. But this is a large engineering lift (persistent threads, two-phase commit on device, or round-trips per refinement level), the workload is branchy and irregular, and it eliminates the clean property you currently have: the tree is built by sound f64 code paths that also define correctness. The recommendation is to keep all mutation on the CPU permanently and treat the GPU as a read-only field oracle + ascent engine. Amdahl then caps end-to-end speedup at the insert path's share — I'd estimate a 3–10× wall-clock win on the 1M benchmark initially, more with pipelining, versus 10–100× on the ascent phase in isolation.

Practical notes

- Stack: wgpu/WGSL (portable, no CUDA lock-in) fits the crate's spirit; the legacy OpenCL wrapper is prior art but its dense-framebuffer model belongs to Argmax2D, not the ADF.
- Determinism: current runs are exactly reproducible (rand_pcg seeds). GPU f32 plus scheduling nondeterminism in dedup ordering breaks that; if reproducibility matters for the art, sort batch results by seed index before the sequential dedup (cheap) and accept value-level drift.
- N dimensions: the future-work item multiplies the case for this — 3D fills scale the ascent workload much faster than the structural one, pushing even more of the runtime into exactly the phase that ports well.

Bottom line: don't chase "everything on GPU." Reify primitives into a shape IR + baked textures, upload the arena as-is, run fill-state-scaled mega-batches of f32 ascents on device, dedup and f64-refine survivors on CPU, keep insertion exactly as it is, and add the one-line liveness recheck so batches can pipeline. That gets most of the theoretical win at a fraction of the redesign, and nothing in the soundness story has to move.
